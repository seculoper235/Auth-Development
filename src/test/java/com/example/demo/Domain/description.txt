 * Test란?
 * Test란 말 그대로 테스트하는 기능이다. 무언가를 본격적으로 살행해보기 이전에 Test를 합으로써, 좀더 안정성 있는 코드를 얻을 수 있다.
 * 누군가 테스트를 설명할 때 오류를 제거해주는 과정이라고 말하지만, 그건 사실 일부분일 뿐이다.
 * Test 과정의 중점은 바로 안정성과 경험이다.
 * 앞으로 만들 기능에 대해, 미리 실패할 코드와 성공할 코드를 작성하여 Test를 완수하게 되면, 그 코드는 하나의 안전한 코드로 의심없이 사용할 수 있다.
 * 누군가 TDD로 개발하게 되면 오히려 시간이 더디게 걸린다고 하지만, 오히려 그렇지 않다.
 * 막무가내로 코드부터 작성하고 "이게 제대로 실행되야 할텐데..."라며 덜덜떨며 실행하는 것보다 훨씬 낫다.
 *
 * 그렇다면 테스트 코드는 어떻게 작성해야 할까? 편리한 라이브러리도 있고 나름의 작성하는 틀도 있겠지만, 결국 경험이 좌우한고 생각한다.
 * 어떤 것이 에러가 떠야하고 어떤 것이 성공해야 하는가는 많이 만들어보고, 이슈도 많이 격어봐야 아는 것이다.
 * 따라서 처음엔 Test 코드를 작성하기 어려운 것은 당연하다고 생각한다.
 *
 *
 * Test의 종류?
 * Test에는 크게 단위/통합/인수 테스트가 있다. 순서대로 점점 스케일이 커지는 방식이며,
 * 단위는 말 그대로 독립적인 하나의 기능을 테스트하는 것이고,
 * 통합은 테스트를 거친 단위들을 합쳐서 실행하더라도 안정적인지를 테스트하고,
 * 인수는 실서버를 돌리기 전에 마지막으로 총체적인 기능(인프라 등)을 테스트하는 과정이다.
 * 여기서는 단위의 기본중의 기본, JPA 테스트에 대해 알아보자
